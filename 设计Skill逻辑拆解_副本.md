# 设计 Skill 逻辑拆解

> 基于 2026-03-29 设计智力建设讨论录音整理
> 核心共识：**先定义问题范围 → 再按技术路线拆子能力 → 每个子能力可独立评审交付**

---

## 一、前置共识（来自录音）

| 共识点 | 录音原话 |
|--------|---------|
| 业务需求 ≠ Skill 规划 | "核心逻辑就是把业务需求跟你的 Skill 规划这两件事情分开" |
| Skill 按技术路线划分，不是按业务场景 | "Scale 本身的技术构建是由技术来区分的，但场景跟技术又是交叉的" |
| 先拆清楚再评审，不能拆不清就开做 | "不能提一个需求叫需求 to UI，这个需求就评不了" |
| 每个 Skill 必须能 2 天内完成一个可交付单元 | "做的每一件事情一定是能在两天内做出来的事情" |
| 竞品对标 = 质量衡量标准 | "最好的衡量交付质量的方法就是判断是否达到了竞品的水平" |
| Skill 分两层：基础 Skill + 业务定制 Skill | "第一层叫做基础式机…第二种叫业务定制式机" |

---

## 二、五大方向定义

录音明确确认的五大方向（4 月 5 日首批交付）：

```
1. UI 生成
2. 图片生成
3. 视频 / 动效生成
4. 文档（含 PPT）
5. 评测
```

---

## 三、逐方向拆解

### 3.1 UI 生成

**拆解维度：交付物类型（原型类 vs 生产类）**

```
UI 生成
├── 原型类（低保真 → 可走查）
│   ├── 单页 HTML（静态展示用）
│   └── 可交互原型（多页流转，如 Figma 可点击态）
│       └── 竞品标杆：Google IDX / v0.dev
│
└── 生产类（直达生产环境代码）
    ├── 移动端
    │   ├── Android（Compose / XML）
    │   └── iOS（SwiftUI / UIKit）
    └── Web 端
        ├── H5 / 落地页（Landing Page）
        │   └── 含营销物料、活动页场景
        ├── B 端控制台
        │   ├── 增删改查型（表单、列表、详情页）
        │   └── 复杂型（数据可视化仪表盘、监控台）
        └── 组件级（Button / Icon 选配等原子能力）
```

**关键决策点（录音提示）：**
- 当前 Focus：**B 端场景**（P2D 直达生产代码）
- 最难定义的问题：输出物是 Figma 格式、HTML、还是 React 组件代码？→ 必须在评审前给答案
- 子能力可独立：找图标 = 一个 Skill（"帮你的 UI 选一个好看的图标都可以做成一个 Skill"）

---

### 3.2 图片生成

**拆解维度：生成技术路线**

```
图片生成
├── 文生图（Text-to-Image）
│   ├── 通用场景（Flux / SDXL 等模型调用）
│   └── 品牌风格场景（含风格锁定、品牌色约束）
│
├── 图生图（Image-to-Image）
│   ├── 风格迁移 / 图片改写
│   └── 局部重绘（Inpainting）
│
├── 合成图（Compositing）
│   ├── 底图生成（如明星合照背景底图）→ 更适合工作流而非 Skill
│   └── 元素拼合（素材 + 版式自动组合）
│
├── SVG / 矢量插画生成
│   └── 竞品标杆：GPT-4o SVG 输出、Midjourney vector
│
└── 配图 / 插画（文案驱动）
    ├── 子能力A：给定配图需求直接生图
    └── 子能力B：从长文中自动判断插图位置 + 生图
        └── 意图识别问题：先路由到技术路线再执行，还是先评估多方案再选最优？
            → 建议：先评估多方案（SVG/生图/图库），再路由子技能
```

**关键决策点：**
- 明星合照底图这类高度定制化需求 → **用工作流解决，不入 Skill 库**
- "它太过于定制化，有些 Skill 都不用推到市场去"

---

### 3.3 视频 / 动效生成

**拆解维度：复杂度分级（L1-L3）× 技术路线**

#### 3.3.1 复杂度分级（王杰视角，供参考）

| 级别 | 描述 | 典型场景 |
|------|------|---------|
| L1 | 微动效 | 背景动效、按钮 hover、PPT 炫发物料 |
| L2 | 独立动效 | Loading 动画、启动屏、品牌 Reveal |
| L3 | UI 层级动效 | 页面转场、复杂交互动效序列 |

#### 3.3.2 技术路线分类（产品视角，拆 Skill 的依据）

```
视频 / 动效生成
├── Remotion（React 驱动视频）
│   └── 适合：数据可视化视频、信息图动画、汇报视频
│
├── HTML + CSS / SVG 动效
│   └── 适合：L1-L2 微动效、H5 内嵌动效
│
├── Lottie / JSON 动效
│   └── 适合：移动端 Loading、App 内动效交付
│       技术路径：生成 Lottie JSON → 客户端渲染
│
└── AI 视频生成（Kling / Sora 等模型）
    └── 适合：营销素材、L3 复杂视频（能力上限待评估）
```

**关键决策点：**
- 驱动 AE（After Effects）直接生成：技术难度高，暂不列入
- 每条技术路线对应一个独立 Skill，**不能合并成"需求 → 视频"一个大 Skill**
- 可行性评审必须在做之前完成（由杨帆 + 子宏评估）

---

### 3.4 文档生成（含 PPT）

**拆解维度：文档类型 × 输出格式**

```
文档生成
├── PPT / 演示文档
│   ├── 品牌屋型（信息架构复杂，可编辑）
│   │   └── 竞品标杆：Skywork Agent（能画品牌屋并可编辑）
│   └── 美学型（视觉表现强，但信息架构简单）
│       └── 竞品参考：张月光系产品
│           → 两条路线选其一作为当前质量标准
│
├── 调研报告 / 分析文档
│   ├── AI 样式调研
│   ├── 素材调研（市场竞品素材扫描）
│   └── 品牌报告（品牌 MR 提效）
│
├── 交互文档（B 端设计规范）
│   └── 可行性待评估（录音：未来形态下意义不大）
│
└── 信息图（Infographic）
    └── 已有基础（infovis 系列 Skill）
```

**关键决策点：**
- PPT 质量标准必须二选一（信息架构完备 OR 美学表现优先），不能两者都要
- "一定要交付特别高可编辑性的 PPT → 通过竞品分析发现没有任何竞品能做到 → 说明现阶段做不到"
- 走查类（设计走查、代码走查）→ 输出物虽是文档，但**属于评测方向**，不属于文档生成

---

### 3.5 评测

**拆解维度：评测对象 × 评测维度**

```
评测
├── UI / 界面评测
│   ├── 可用性评测（布局、层级、交互合理性）
│   ├── 视觉规范评测（设计 token、间距、色彩规范）
│   └── 竞品对标评测
│
├── 图片 / 视觉质量评测
│   ├── 美学评测（AQE：视觉和谐度、色彩美学等）
│   └── 品牌合规评测
│
├── 视频 / 动效评测
│   └── 流畅度、节奏感、品牌一致性
│
├── 文档质量评测
│   └── PPT 结构完整性、信息密度、视觉规范
│
└── 设计走查（Design Review）
    ├── 自动化走查（设计稿 → 代码还原度）
    └── 人工辅助走查（People-in-Loop 机制）
```

**关键决策点：**
- 录音原话："评测美学评测，你俩都得拆，评测不一样的"
- 每个生成方向对应一个评测 Skill（UI 生成 → UI 评测，图片生成 → 图片质量评测）
- 评测 Skill 定位：**既是质量门控，也是自进化反馈入口**

---

## 四、Skill 分层架构

```
设计 Skill 体系
│
├── 基础 Skill（通用，面向所有用户，可推市场）
│   ├── UI 生成类：B 端控制台生成、H5 落地页生成、可交互原型生成...
│   ├── 图片生成类：文生图、SVG 插画生成、配图生成...
│   ├── 视频类：Remotion 数据视频、Lottie 动效生成...
│   ├── 文档类：PPT 生成、调研报告生成、信息图生成...
│   └── 评测类：UI 评测、图片美学评测、PPT 质量评测...
│
└── 业务定制 Skill（高度定制，不推市场，仅供特定业务用）
    └── 如：明星合照底图生成（与具体业务场景深度绑定）
```

---

## 五、工作流 SOP（录音达成共识）

```
1. 拆解
   └── 每个方向拆成可独立评审的最小 Skill 单元
       拆不清楚 → 找杨帆协助拆（技术视角补充）

2. 评审（做之前必做）
   └── 产品提需求（你） → 技术评估可行性（杨帆 + 子宏）
       → 不可行直接砍，不浪费两天

3. 对标竞品
   └── 每个 Skill 找到对应竞品 → 明确"达到竞品水平"= 合格

4. 交付 & 入库
   ├── 业务需求做完 → 选择性入通用库
   └── 业务过程中发现可复用模块 → 提议公共化

5. 评测闭环
   └── 每个生成 Skill 对应一个评测 Skill，形成生成-评测自进化循环
```

## 六、4 月 5 日交付目标

每个方向交付一个**完整定义的示范 Skill**（含：技术路线 + 输入输出定义 + 质量标准 + 竞品对标）：

| 方向 | 示范 Skill（建议） | 质量标准锚点 |
|------|------------------|------------|
| UI 生成 | B 端增删改查控制台生成 | 与 v0.dev / Vercel AI 对标 |
| 图片生成 | 文案驱动配图生成（SVG 路线） | 与 GPT-4o SVG 输出对标 |
| 视频 / 动效 | HTML+CSS 微动效生成（L1） | 与 Framer Motion 示例对标 |
| 文档（含 PPT） | 品牌屋型 PPT 生成 | 与 Skywork Agent 对标 |
| 评测 | UI 可用性评测 | 内部评测标准 + TRAIL 模型 |
