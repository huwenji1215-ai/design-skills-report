设计 Skill 三件事

回答三个核心问题：做什么（边界与优先级）、怎么做（工程化路径）、做到什么算完成（可量化验收）

📦 业务需求 15 项 ⚡ P0 立刻启动 3 项 🔧 P1 近期建设 5 项 ❓ 待澄清 2 项
前置诊断：为什么需要重新定义
当前 skill 体系是以技术模块为中心组织的 — 按「视觉生产/品牌/动效/设计系统」分域，每个节点挂功能列表。这回答的是「有什么工具」，但没有回答三件更本质的事：

用户意图是什么 × 做到什么程度值得做 × 做完的标准是什么

三件事框架强制把这三个维度显式化，让 skill 从「功能清单」升级为「可交付的产品能力」。

  • Skill 调用链路原子化拆解

第二件事
如何提升 Skill 的工程化程度
意图识别前置 + 原子单元拆解 + 工具链更新
2.1 用户意图识别和澄清
当前问题：大多数 skill 的 description 是「能力描述」，不是「意图描述」。用户说「帮我弄个动效」，skill 体系不知道这是 GSAP 滚动视差、Lottie JSON、AE 粒子复现还是 CSS 过渡。每个 skill 入口必须有意图识别树。
示例：「动效」需求的意图识别树
# 用户说「动效」→ 触发意图识别 ├── 是否已有设计稿/原型？ │ ├── 有 → 「还原型」 → 匹配 GSAP / CSS Animation │ └── 无 → 「创作型」 → 进一步问风格/品牌调性 ├── 目标平台？ │ ├── Web → GSAP / CSS │ ├── App → Lottie JSON │ └── 视频 → AE 脚本 / Remotion └── 复杂度？ ├── 简单过渡 → 直接输出代码 └── 高级特效 → motion-effect-generator → AE 粒子复现链路
落地规范：每个设计 skill 在 SKILL.md 中必须包含 intent_clarification 前置区块，用 YAML 定义意图树。触发 skill 时先走意图识别，再进入能力执行。
2.2 Skill 调用链路原子化拆解
示例：「交互文档生成」的原子链路拆解（3个原子单元，Input → Process → Output 完全显式）
原子 Skill 1 — intent-extractor（意图提取）
IN: 设计稿截图 / Figma URL / 用户描述 PROCESS: 提取组件列表、交互状态、流转逻辑 OUT: 结构化 YAML（组件树 + 状态机）
原子 Skill 2 — interaction-doc-writer（文档写作）
IN: 结构化 YAML PROCESS: 套用交互文档模板，填充描述 OUT: .docx / Markdown 交互说明文档
原子 Skill 3 — design-qa-checker（完整性检查）
IN: 生成的交互文档 PROCESS: 对照 Checklist（异常态/空态/动效描述覆盖） OUT: 缺项报告 + 补全建议
2.3 工具链更新：Figma MCP 替代 Stitch
对比维度Stitch（Google）Figma MCP（当前主流）
归属Google 独立产品Figma 官方 MCP Server
核心能力草图/文字 → UI 组件生成AI Agent 直接读取/操作真实 Figma 文件结构
设计→代码协同单向生成，与 Figma 无原生连接读取节点树、组件属性、Token、约束，双向同步
在设计工作流中的定位独立生成工具，不在 Figma 工作流内Figma 工作流的 AI 扩展层，无缝集成
当前成熟度Beta，功能受限正式可用，Cursor/Claude Code 均支持
在全景图中的处理 移除 Stitch 引用 nd-figma-review 更新为「Figma MCP + AI Agent」
第三件事
如何判断 Skill 的完成度
每个原子 Skill 解决什么问题 + 三层可量化验收体系
3.1 每个原子 Skill 必须回答的五个问题
反例（当前写法）：「智能可视化图片生成器，自动理解意图、选择设计方式、生成美学达标图片」— 这回答不了「做完了没有」。
字段说明示例（visual-creator）
解决的核心问题用户之前手工要花多少时间设计师手工选图表类型+调色+排版，平均 45 min/张
用户场景谁在什么情境下调用数据分析师拿到数据后，需要做汇报 PPT 配图
精确 Input接受什么格式数据表 + 意图描述（「展示各部门Q1对比」）
精确 Output输出什么格式PNG图片 + ECharts 代码 + 设计说明
通过标准（量化）什么分数算合格infovis-evaluator 评分 ≥ 8.5 / 10
3.2 三层验收体系
⚙️
Layer A — 功能验收
Skill 能跑通
5 种典型 case 能准确路由到正确处理路径
输出格式符合定义，无幻觉字段
输入不合法时有合理提示，不直接报错
意图识别分支均有 fallback 处理
✅
Layer B — 质量验收
Skill 有用
视觉生成类：design-evaluator ≥ 8.5 分
图表/InfoVis 类：infovis-evaluator ≥ 9.0 分
文档生成类：结构完整度 ≥ 90%（必填项无空缺）
评测 Skill：与人工评审一致性 ≥ 85%
🚀
Layer C — 效率验收
Skill 值得用
调用时长 + 修改时长 < 人工时长的 50%
用户平均修改次数 ≤ 2 次可接受
实际使用过的设计师复用率 ≥ 60%
用户主动推荐给同事率 ≥ 40%

第一件事
战略层
做哪些、不做哪些，资源如何分配。P0 三件都是包装现有能力，立刻可启动。
第二件事
工程层
怎么把能力封装成可复用、可组合的单元。意图识别前置 + 原子拆解是核心。
第三件事
质量层
做出来的东西有没有真正解决问题。三层验收保证功能、质量、效率缺一不可。
