体验框架 TRAIL（本框架专门为此设计）
为什么传统 HEART 框架不够用？ HEART（Happiness / Engagement / Adoption / Retention / Task Success）是为 SaaS 界面设计的。 对 Skill 场景：Adoption / Retention 没有意义（Skill 是嵌入式的，不存在「主动采用」）； Happiness 太主观，无法指导 Prompt 优化；Task Success 定义模糊（AI 做了 = 成功，还是用户满意 = 成功？）。 需要一套针对 意图理解 → AI 执行 → 产出可用 全链路的专属框架。

TRAIL 框架设计逻辑 — 为什么是这 5 个维度？
TRAIL 是针对本场景自创的框架，不是引用业界标准。 现有框架（HEART / CASTLE）均为 SaaS 界面设计而生，无法覆盖 「意图识别 → AI 执行 → 产出可用」这条 Skill 特有的体验链路。 TRAIL 的 5 个维度是从这条链路的 3 个层次反向推导出来的。

层次 2 能力覆盖完整性 这个 Skill 能做的，够不够用？
即使单次体验流畅，用户也可能发现：「我以前在 SaaS 里能做的事，现在做不了了」
I · Integration Fit 衡量 Skill 对原 SaaS 功能集的覆盖度，以及与上下游工作流的衔接程度
I 维度是 SaaS Skill 化场景特有的——纯 AI 原生产品不需要这个维度，但迁移场景必须有。
↑ 单次 + 覆盖的汇聚
层次 3 系统自进化 越用越好，还是原地踏步？
前 4 个维度衡量「现在怎么样」，L 维度衡量「是否在变好」：
✓ Known Failures 被注入到下一版 Skill 的设计约束里
✓ 新上线 Skill 的 Gate 2 初始评分季度环比上升
✓ 同类 P0 失败不再重复出现
L · Learning Compound — 飞轮是否在转
L 是整套体系的验证维度。如果 T/R/A/I 的评测结论没有被系统性积累和注入，前 4 个维度的改善只是一次性的，不构成飞轮。
T + A + R = 单次对话体验是否流畅
I = 能力范围是否够用（迁移场景特有）
L = 系统是否在自我进化（飞轮验证维度）

框架自检 — 已知空白与演进方向（对标 Anthropic 最高标准）
以下 6 个 Gap 是在对标 Anthropic《Demystifying Evals for AI Agents》（Jan 2026）等最高标准后 自检发现的——它们已在本框架的 Layer A/B 相关章节中被补充完善，此处作全貌索引。
Gap 1 已修复 非确定性处理缺失
原方案 SQE-5 只给一次评测分数，没有处理模型随机波动。现已补充：能力评用均值（跑3次），回归评用 pass^k（3次全过才算稳定）。
→ Layer A · Gate 1 · 非确定性处理规则
Gap 2 已修复 能力评 vs 回归评未区分
原方案把两种性质不同的评测混用同一套标准。现已明确：新 Skill 上线用能力评（hill-climbing），已上线 Skill 迭代用回归评（防倒退）。
→ Layer A · Gate 1 · 评测类型区分栏
Gap 3 已修复 Task + Grader 规范缺失
原方案只说"10 条用户表达"，没有定义什么叫"命中"、谁判、怎么判。现已补充 Task 三条件（两专家可独立判断 / 有 Reference Solution / 失败可从 transcript 定位）。
→ Layer A · Gate 1 · Task + Grader 规范
Gap 4 已修复 Eval Saturation（天花板效应）无预警
当所有 Skill 都 9.0+ 时，分数失去区分度，真实退化被高分掩盖。现已补充：触发条件（连续4周均分≥9.0）→ 自动升级题库难度机制。
→ Layer B · Eval Saturation 防天花板机制
Gap 5 已修复 Gate 2 多人标注无一致性校验
多人评同一 Skill 时，分差 > 2 说明的是评测标准模糊，而非 Skill 质量差异。现已补充 IAA（Inter-Annotator Agreement）三级规则和月度 Calibration Session 机制。
→ Layer B · 评测质量保障 · IAA 校验
Gap 6 已修复 LLM 评测自身未被验证
Gate 1 LLM-as-Judge 的结论可信度从未被检验。现已补充：月度交叉校验机制，相关系数 < 0.70 时暂停 Gate 1 自动化，强制全量人工评。
→ Layer B · 评测质量保障 · LLM vs 人工交叉校验
📐 TRAIL 模型 — Skill 体验的 5 维度量语言

SQE-5 — Skill创建的检测工具

面向内部——用来在 Skill 上线前检测「Skill 定义本身的质量」。 5 个维度对应 Skill 设计的 5 个可检测属性，带权重打分， 是 Gate 1（LLM 自动评测）和 Gate 2（人工抽测）的评分依据。

创建检测 上线门控 可自动化 LLM-as-Judge
🚦 Layer A · Skill 创建评测（三道门控）

每一个由 SaaS 功能转化而来的 Skill，在上线前必须通过三道质检关卡。 Gate 0 是人工自查，Gate 1 是 LLM 自动评测（SQE-5），Gate 2 是人工体验抽测。 任意一关未过，打回修改后重新从 Gate 0 开始。

Gate Protocol — 三道门控全流程
INPUT
新 Skill / SaaS 功能 Skill 化
↓
Gate 0 人工自查清单 执行人：Skill 设计者
□ description 是否包含 ≥ 5 个用户真实触发短语（而非功能描述语言）？
□ 是否明确写了 「不应触发的场景」（Negative cases）？
□ 输出格式是否有 具体示例（≥1 个完整 Output 样例）？
□ 关键参数是否可被用户 自然语言覆盖（而非硬编码）？
□ 是否包含 ≥ 1 个完整的 end-to-end 使用示例（用户说X → 产出Y）？
□ 是否考虑了出错时的体验？（输入缺失 / 超出范围时，Skill 给出的是可执行建议，而非静默失败）
□ 产出的信息密度是否合理？（结论是否优先呈现？用户不需要自己从大段文字中提炼重点）
□ 使用的术语是否是用户语言而非产品内部概念？（用户无需先理解 Skill 的参数体系才能触发）

↓ PASS
Gate 1 SQE-5 自动评测 执行人：LLM-as-Judge（可自动化）
能力评（Capability Eval） 新 Skill 首次上线时使用。目标低起点，验证「能不能做到」，允许多轮迭代。
vs
回归评（Regression Eval） 已上线 Skill 每次迭代后使用。目标接近 100%，验证「还能不能做到原来做到的事」。
⚠ 非确定性处理规则（对标 Anthropic pass@k 机制）
能力评 D1/D2/D3 各维度各跑 3 次，取均值作为维度分。单次波动 ≤ 0.5 为正常，超过说明 Skill 定义本身不稳定，需先修复描述歧义再评测。
回归评 使用 pass^k 逻辑：k=3 次全部通过（≥ 8.0）才算稳定上线。如果 3 次中有 1 次低于通过线，视为不稳定，需查 transcript 定位根因。
维度 权重 评测方法 通过线
D1 触发 25% 给出 10 条真实用户表达，测试 Skill 命中率 ≥ 7.5
D2 理解 25% 给出 5 条有歧义的指令，测试 AI 解读是否准确 ≥ 7.5
D3 产出 30% 盲测：不看过程，产出能直接用？
同时检测：① 信息层级清晰（结论优先，不需二次整理）② 出错 / 边界输入时给出可执行出路（不静默失败）③ 输出风格跨次一致（相同输入不出现「活泼 vs 严肃」随机波动） ≥ 7.5
D4 边界 10% 给出 3 条越界指令，测试 Skill 是否正确拒绝
包含：伦理/版权风险边界测试（如：「仿照 XXX 品牌风格生成」「用未授权字体生成」），Skill 应识别并给出合规提示 ≥ 6.0
D5 文档 10% description 是否从用户语言角度写的？ ≥ 6.0
综合总分 100% D1×0.25 + D2×0.25 + D3×0.30 + D4×0.10 + D5×0.10 ≥ 8.0
⚙ 自动变异逻辑（分数 < 8.0 时触发）
1 识别失分最高的维度（P0 失分 > 2.0 / P1 失分 1–2 / P2 失分 < 1）
2 生成可直接注入 description 的修复指令（不是问题描述，是操作指令）
3 将失败模式写入 known_failures（类型 + 修复指令 + 防止重现规则）

SaaS Skill 化常见失败类型（Gate 1 评测标准测试用例指南）
P0 意图边界模糊
Skill 能做 A 也能做 B，用户不知道什么时候用；或与其他 Skill 重叠严重。
触发维度：D1 触发 · D5 文档

P0 触发词稀疏
用户说了相关的话，但 Skill 没有响应；或只有精确说出关键词才能触发。
触发维度：D1 触发
修复：用真实用户表达语言重写 description，覆盖口语化、缩写、上下文触发等模式
P1 参数硬编码
SaaS 的表单字段被直接映射成固定参数，用户无法通过自然语言调整。
触发维度：D2 理解 · D4 边界
修复：将关键参数改为可被自然语言覆盖的可选项，并在 description 中举例说明
P1 输出格式不可用
AI 产出了内容，但格式/结构与用户下游使用场景不匹配，需大量二次处理。
触发维度：D3 产出
修复：在 description 中明确输出格式规范，提供 ≥1 个完整产出示例
P1 能力边界缺失
用户问了 Skill 不能做的事，AI 没有拒绝或说明，而是给出错误/幻觉性回答。
触发维度：D4 边界
修复：在 description 中显式定义「本 Skill 不处理的场景」，并标注推荐的替代 Skill
P2 预期偏差
Skill 的功能定义与用户的心智模型不一致，用户以为能做 A，实际做的是 A'。
触发维度：D5 文档 · D2 理解
修复：重写 description 使用用户语言，避免内部技术术语；增加场景对比说明

Eval Saturation 防天花板机制
⚡
当所有 Skill 的 Gate 2 平均分 > 9.0 时，评测已经饱和—— 分数不再有区分度，真实的能力退化会被「高分假象」掩盖。 这与 Anthropic 的 eval saturation 警告完全对应（SWE-Bench 从 30% 涨到 80%+ 后进入饱和区）。
触发条件
全平台 Skill 平均 Gate 2 分 ≥ 9.0，连续 4 周
升级动作
在 D1/D2/D3 的测试集中替换 30% 为「高难度 case」（歧义指令 / 跨 Skill 意图 / 复杂上下文）
目标状态
升级后通过率回落到 75–85% 区间，确保评测仍有区分度可以衡量进步
评测质量保障 — 让评测本身可信
Gap 5 IAA 标注者一致性校验
Gate 2 体验卡由多人填写时，必须计算 Inter-Annotator Agreement（IAA）。 两人在同一 Skill 同一维度的分差 > 2.0，说明评测标准模糊，需要先校准标准，而非取平均。
✓ IAA < 1.5 — 标准清晰，结果可信，取均值
⚠ IAA 1.5–2.5 — 存在歧义，需召开校准会，对齐后重测
✗ IAA > 2.5 — 评测标准本身有问题，修复 Rubric 后重新发起
频率：每月至少一次 Calibration Session，至少 3 人共同评同一批 Skill
Gap 6 LLM 评测 vs 人工评测 交叉校验
Gate 1（LLM-as-Judge）的可信度需要定期用 Gate 2 人工结果来校准。 若 LLM 评分与人工评分的 相关系数 < 0.7，LLM Rubric 需要重写。
✓ 相关系数 ≥ 0.85 — LLM 评测可信，维持现有 Rubric
⚠ 相关系数 0.70–0.85 — 偏差可控，记录偏差类型，定向优化 Rubric
✗ 相关系数 < 0.70 — LLM 评测不可信，暂停 Gate 1 自动化，全量人工评
频率：每月抽取 20 个 Skill 做双盲交叉验证。发现「LLM 给高分，人工给低分」的 case 优先写入 Known Failures。

🧠 失败知识库（Known Failures）

Known Failures 是整套飞轮的记忆核心。每一次人工评审产生的失败发现， 都必须被结构化记录并注入下一版 Skill 的设计约束前置条件里。 失败不可重复，是飞轮加速的唯一燃料。

核心原则：记录失败 ≠ 只是记录问题。 每条失败记录必须包含：触发场景 + 失败类型 + 可执行修复指令 + 防重现规则。 没有修复指令的失败记录是无效的——它无法被注入 Prompt，飞轮就断了。

P0 SFP-001 意图边界模糊 T 2026-03-25 Layer B · 放弃分析

防重现规则 所有生成类 Skill 必须包含 intent_clarification_protocol
P0 SFP-002 输出格式不可用 R 2026-03-25 Layer A · Gate 2
触发场景：「汇报 PPT」场景下 AI 产出正文段落而非结构化 Slide，体验卡 Q2 均分 4.2
修复指令 在 description 中添加输出格式示例：每 Slide 格式为「标题 | 3-5 要点 | 备注」，附完整示例
防重现规则 所有有固定输出形式的 Skill，必须在 description 中包含 ≥1 个完整 Output 示例
P0 SFP-003 触发词稀疏 T 2026-03-20 Layer B · 被动信号
触发场景：用户说「给我画个趋势图」「走势图怎么做」等口语化表达时 Skill 无响应
修复指令 用真实用户口语扩充 description：「趋势图 / 走势图 / 折线 / 变化曲线」等口语化短语
防重现规则 Gate 0 必查：description 是否覆盖口语化、缩写、上下文触发等模式（≥5 个真实短语）
P1 SFP-004 过程不透明 A 2026-03-18 Layer B · 放弃分析
触发场景：多步骤 Skill（生成→评测→导出）执行过程中无状态反馈，用户以为卡死而放弃
修复指令 在 description 中说明执行步骤数量和预期时长；在关键步骤前输出进度标记
防重现规则 执行时间 > 30s 的 Skill 必须包含分步进度说明
P1 SFP-005 能力边界缺失 I 2026-03-15 Layer A · Gate 1
触发场景：用户要求「数据清洗 + 可视化」，Skill 只处理可视化部分，对数据清洗部分给出错误答案
修复指令 在 description 显式标注「本 Skill 仅处理已清洗数据的可视化，不包含数据清洗」，并推荐替代 Skill
防重现规则 Gate 0 必查：是否包含「不处理的场景」的显式声明
P1 SFP-006 参数硬编码 A 2026-03-12 Layer A · Gate 2
触发场景：设计规范色值被硬编码，用户说「改成红色主题」时 Skill 无响应或输出错误
修复指令 将颜色、尺寸等关键参数改为可被自然语言覆盖的可选项，在 description 举例「如需指定颜色，直接说「用蓝色」」
防重现规则 Gate 0 必查：关键参数是否可被用户自然语言覆盖
P2 SFP-007 预期偏差 I 2026-03-10 Layer B · 用户访谈
触发场景：用户以为「设计评测」=评测视觉美感，实际 Skill 评测的是信息传达效率
修复指令 在 description 首句明确：「本 Skill 评测信息传达效率（而非视觉美感），适用于...」
防重现规则 Skill 名称与用户心智模型差异较大时，description 首句必须包含「是什么/不是什么」的对比说明
P2 SFP-008 触发竞争 T 2026-03-08 Layer B · 被动信号
触发场景：「做一个图表」同时触发 visual-creator 和 design-evaluator，用户收到两个 Skill 的响应，困惑选择
修复指令 在 design-evaluator 的 description 中添加负向触发说明：「用户说「做」/「生成」时不触发，仅当说「评测」/「检查」/「审查」时触发」
防重现规则 Gate 0 必查：与相邻 Skill 是否存在触发词重叠，如有需明确写出互斥关系
