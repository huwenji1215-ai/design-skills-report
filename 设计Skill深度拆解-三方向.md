# 设计 Skill 深度拆解：五大方向全景

> 版本：v2.0 | 2026-03-30
> 拆解原则：按技术路线划分，每个 Skill 可独立评审，2天内可交付一个单元
> 覆盖方向：UI生成 × 图片生成 × 视频动效 × 文档生成 × 评测

---

## 方向一：UI 生成

### 整体结构

```
UI 生成
├── A. 原型类
│   ├── A1. 单页 HTML 原型
│   └── A2. 可交互多页原型
│
└── B. 生产类
    ├── B1. B端增删改查控制台        ← 当前优先级 P0
    ├── B2. B端数据可视化仪表盘       ← P1
    ├── B3. H5 / 营销落地页          ← P1
    ├── B4. 移动端（Android/iOS）    ← P2
    └── B5. 组件原子能力（图标选配等）← P1（轻量高频）
```

#### Skill UI-B1：B端增删改查控制台生成

**一句话定义**
输入产品需求描述，直接输出可运行的 B 端管理后台页面代码（含列表、表单、详情三件套）。

**技术路线**
LLM 代码生成 → React + Tailwind CSS + shadcn/ui 组件库

**输入**
```
- 业务实体描述（如：用户管理、订单管理）
- 字段列表（可选，未提供则 AI 自推断）
- 交互需求（可选，如：可筛选、可批量操作）
- 输出格式：React 组件代码 / 纯 HTML
```

**输出**
```
- 列表页（含分页、搜索、筛选）
- 新增/编辑表单页（含字段校验）
- 详情页（可选）
- 可直接运行的完整代码（含样式，无需额外依赖配置）
```

**质量标准（竞品对标：v0.dev）**
| 维度 | 标准 |
|------|------|
| 视觉完整度 | 与 v0.dev 生成质量持平（布局合理、无明显错位） |
| 代码可用性 | 直接粘贴到 React 项目可运行，0 报错 |
| 交互完整性 | 列表分页、表单提交、错误提示等基础交互覆盖 |
| 响应式 | 最低支持 1024px 宽屏 |

**边界（不做什么）**
- ✗ 不生成后端接口（只做前端 Mock 数据）
- ✗ 不支持 Vue/Angular（锁定 React 生态）
- ✗ 不做复杂权限体系（超出单页范围）

**可行性评级：⭐⭐⭐⭐⭐ 高**
已有成熟技术路线（LLM + shadcn/ui），v0.dev 已验证可行

#### Skill UI-B2：B端数据可视化仪表盘生成

**一句话定义**
输入数据结构或业务场景，输出含图表的数据监控/分析仪表盘页面代码。

**技术路线**
LLM 代码生成 → React + ECharts / Recharts + Tailwind CSS

**输入**
```
- 数据描述（字段名 + 数据类型，如 JSON schema）
- 仪表盘用途（如：运营监控、销售分析）
- 图表偏好（可选，如：折线图 + 柱状图 + 饼图）
```

**输出**
```
- 完整仪表盘页面代码
- 含 Mock 数据（可替换为真实接口）
- 图表类型：折线 / 柱状 / 饼图 / 数字卡片
```

**质量标准（竞品对标：Vercel AI + ECharts 官方示例）**
| 维度 | 标准 |
|------|------|
| 图表准确性 | 图表类型与数据类型匹配，无误用 |
| 视觉层级 | 数字卡片 + 主图 + 辅助图的信息层级清晰 |
| 代码可用性 | 直接运行可展示完整仪表盘 |

**边界**
- ✗ 不做实时数据接入（Mock 数据只）
- ✗ 复杂 3D 图表暂不支持（ECharts 3D 系列 → 后续迭代）

**可行性评级：⭐⭐⭐⭐ 较高**（图表类型多，需明确约束范围）

#### Skill UI-A1：单页 HTML 原型生成

**一句话定义**
输入需求描述，快速生成一个可浏览器打开的单页静态 HTML 原型，用于走查和沟通。

**技术路线**
LLM 代码生成 → 纯 HTML + CSS + 少量内联 JS

**输入**
```
- 页面功能描述（自然语言）
- 参考截图（可选）
- 风格偏好（可选，如：深色 / 轻量 / 企业级）
```

**输出**
```
- 单个 .html 文件，浏览器直接打开
- 不依赖外部框架，可独立运行
```

**质量标准（竞品对标：Claude Artifacts / ChatGPT Canvas）**
| 维度 | 标准 |
|------|------|
| 布局还原度 | 主要功能区域位置正确，无明显走位 |
| 视觉可用 | 能让设计师/PM 看懂并走查，不要求像素级精准 |
| 独立运行 | 单文件，无外部依赖，双击即可打开 |

**可行性评级：⭐⭐⭐⭐⭐ 高**（最成熟的 LLM 生成场景之一）

#### Skill UI-B5：UI 图标智能选配

**一句话定义**

```
- 风格偏好（Outline / Filled / Duotone）
```

**输出**
```
- 3-5 个候选图标（含预览 + SVG 代码）
- 推荐理由
- 可直接复制使用的 React/HTML 代码片段
```

**质量标准**
| 维度 | 标准 |
|------|------|
| 语义匹配度 | Top-1 推荐命中率 ≥ 85% |
| 多样性 | 提供 ≥ 3 个风格/形态各异的候选 |

**可行性评级：⭐⭐⭐⭐⭐ 高**（轻量，ROI 极高）

#### Skill UI-A2：可交互多页原型

**一句话定义**
输入产品功能描述，生成支持页面跳转和状态流转的可交互多页原型，适用于用户测试、产品走查和开发需求确认。

**技术路线**
LLM 代码生成 → React Router + 状态管理（Zustand/useState）→ 单仓库多页面打包（Vite）

**输入**
```
- 产品描述（功能模块 + 主要用户流程）
- 页面列表（可选，未提供则 AI 推断）
- 交互重点（如：登录流程 / 支付流程 / 核心操作路径）
- 技术框架（可选，默认 React）
```

**输出**
```
- 多页面 React 应用代码（可本地运行）
- 页面间路由跳转可用
- 核心状态（登录态、表单提交结果等）可模拟
- README：如何本地启动
```

**与 UI-A1 的区别**
| | UI-A1 单页原型 | UI-A2 多页原型 |
|---|---|---|
| 复杂度 | 单文件，双击打开 | 需本地 npm 环境 |
| 适用场景 | 快速走查，1-2 分钟看完 | 用户测试，完整流程体验 |
| 交互深度 | 静态展示为主 | 路由跳转 + 状态流转 |
| 交付时间 | 30 分钟内 | 2-4 小时 |

**质量标准（竞品对标：Figma Make / Bolt.new）**
| 维度 | 标准 |
|------|------|
| 页面完整性 | 核心流程页面 100% 覆盖，无死链 |
| 交互可用 | 主流程操作无报错，表单可填写提交 |
| 可运行性 | `npm install && npm run dev` 一键启动 |

**边界**
- ✗ 不做复杂动效（动效 → VID-F1）
- ✗ 不接真实接口（Mock 数据模拟）
- ✗ 不做复杂权限体系

**可行性评级：⭐⭐⭐⭐ 较高**（React Router 生成成熟，多页面状态需模板化沉淀）

#### Skill UI-B4：移动端 UI 代码生成（Android/iOS）

**一句话定义**
输入移动端产品描述，生成可在 React Native / Expo 环境运行的移动端 UI 代码，适用于产品原型验证和功能演示。

**技术路线**
LLM 代码生成 → React Native + Expo + NativeWind（Tailwind 移动端适配）

**输入**
```
- 功能描述（如：「用户个人中心页」「订单列表+详情」）
- 平台（iOS / Android / 两端通用，默认通用）
- 风格参考（可选，如：微信风格 / 抖音风格）
```

**输出**
```
- React Native 组件代码（.tsx）
- 可在 Expo Go App 扫码预览
- 适配主流机型（iPhone 14 / Pixel 7）
```

**技术选型说明**
| 方案 | 适用 | 特点 |
|------|------|------|
| **React Native + Expo（推荐）** | 原型 + 生产 | 热更新，Expo Go 直接预览 |
| Flutter | 性能敏感场景 | LLM 生成质量稍低 |
| 纯 H5 套壳 | 快速验证 | 体验有限 |

**质量标准（竞品对标：Bolt.new 移动端 / Expo Snack）**
| 维度 | 标准 |
|------|------|
| 可运行性 | Expo Go 扫码可运行，无崩溃 |
| 视觉完整 | 组件样式正确，间距符合移动端规范（8dp 系统） |
| 适配性 | iPhone SE（小屏）和 iPhone 14 Pro Max（大屏）均可用 |

**边界**
- ✗ 不做原生模块（相机/传感器等 → 需单独封装）
- ✗ 不接真实接口（Mock 数据）
- ✗ 不保证 App Store 审核通过（原型用途）

**当前限制**：React Native 代码生成质量低于 Web，建议先做 PoC 评估 LLM 输出质量

**可行性评级：⭐⭐⭐ 中等**（LLM 移动端代码质量需工程评审，建议先 PoC）

## 方向二：图片生成

### 整体结构

**图片生成整体结构**

> ⚠️ InfoVis（信息可视化）是独立路线，不属于「图片生成」范畴，是数据/结构/关系的可视化表达。产品目标②「能做信息图」的核心载体。技术路线三选一：SK1 LLM→SVG/Canvas | SK2 ECharts/D3 | SK3 Figma→PNG渲染

```
图片生成
├── C. 技术路线：LLM 直出代码型
│   └── C1. SVG 矢量插画生成           ← 当前优先级 P0 🔴（产品目标②）
│
├── D. 技术路线：模型调用型（Diffusion / 多模态）
│   ├── D1. 文生图（通用）              ← P1（首选 Gemini 2.5 Flash Nano Banana）
│   ├── D2. 文生图（品牌风格锁定）       ← P2
│   ├── D3. 图生图（风格迁移/改写）      ← P1
│   └── D4. 局部重绘（Inpainting）     ← P2
│
├── E. 技术路线：文案驱动型（上层路由，调度 C/D/InfoVis）
│   ├── E1. 指定配图需求生成图          ← P1
│   └── E2. 长文自动插图（位置+生成）    ← P2
│
├── F. 技术路线：运营裂变型（基于已有图片加工）
│   └── F1. 图片裂变（多尺寸/多平台适配） ← P1
│
└── InfoVis. 信息可视化【独立路线，产品目标②核心】
    ├── IV-A5. 信息图海报（叙事型）       ← P0 🔴（产品目标②首要）
    ├── IV-A1. 数据关系图（柱/折/地图）   ← P1
    ├── IV-A2. 结构层级图（树/思维导图）  ← P1
    ├── IV-A3. 时序流程图（时间轴/桑基图） ← P1
    └── IV-A4. 比较对比图（雷达/热力图）  ← P1
```

#### Skill IMG-C1：SVG 矢量插画生成

**一句话定义**
输入主题描述，直接生成可编辑的 SVG 矢量插画代码，适用于 UI 配图、空状态页、品牌图形。

**技术路线**
LLM 直接输出 SVG 代码（GPT-4o / Claude 3.5 Sonnet）

**输入**
```
- 主题描述（如：「空购物车」、「上传成功」、「团队协作」）
- 风格偏好（可选，如：Flat / Line / Isometric / 3D 感）
- 色板（可选，如：品牌主色 #1677FF）
- 尺寸（可选，默认 400x300）
```

**输出**
```
- 完整 SVG 代码（可直接嵌入 HTML/React）
- 可在 Figma/Illustrator 中进一步编辑
- 文件大小通常 < 10KB
```

**质量标准（竞品对标：GPT-4o SVG 输出）**
| 维度 | 标准 |
|------|------|
| 语义准确性 | 图形内容与主题描述匹配，一眼可识别 |
| 代码合法性 | 标准 SVG 语法，浏览器直接渲染无报错 |
| 视觉质量 | 具备基本美学（构图平衡、颜色协调），不是纯线框 |
| 可编辑性 | SVG 元素有语义化命名，便于后续调整 |

**意图路由设计**
```
用户输入 → 意图判断层
├── 需要写实感 → 路由到 D1（文生图）
├── 需要可编辑矢量 → 路由到 C1（SVG 生成）← 本 Skill
└── 有现成图改风格 → 路由到 D3（图生图）
```

**边界**
- ✗ 不做照片级写实（那是 Diffusion 的工作）
- ✗ 复杂 3D 透视插画效果有限（Flat/Line 风格为主）
- ✗ 不保证每次生成一致（非精确复现场景）

**可行性评级：⭐⭐⭐⭐⭐ 高**
GPT-4o 已验证，Claude 3.5 SVG 能力强

#### Skill IMG-D1：文生图（通用场景）

**一句话定义**
输入文本描述，调用 AI 图像生成模型，产出适用于设计场景的高质量图片。

**技术路线**
模型调用 → GPT-4o Image / Flux.1 / Stable Diffusion

**模型选择策略**
| 场景 | 推荐模型 | 原因 |
|------|---------|------|
| 综合质量最强（2026 Q1 基准） | **Gemini 2.5 Flash Image（Nano Banana）** | 当前业内文生图综合评分第一 |
| 需要精确文字渲染（含文字的图） | GPT-4o Image | 文字渲染能力强，指令遵循精准 |
| 需要高度写实（产品/人物/场景） | Flux.1 Dev | 开源最佳写实效果 |
| 需要快速出图迭代 | Flux.1 Schnell | 速度优先 |
| 有版权合规要求 | Adobe Firefly | 商用授权安全 |

**输入**
```
- 图片描述（支持中文）
- 尺寸比例（1:1 / 16:9 / 9:16，默认 1:1）
- 风格关键词（可选）
- 参考图（可选，用于风格参考）
```

**输出**
```
- 图片文件（PNG/WebP）
- 分辨率：≥ 1024x1024
- 自动附带：生成提示词（便于迭代）
```

**质量标准（竞品对标：Gemini 2.5 Flash Image / Midjourney v6）**
| 维度 | 标准 |
|------|------|
| 主体清晰度 | 主体物体边界清晰，无明显畸变 |
| 提示词遵循 | 关键元素出现率 ≥ 90% |
| 视觉美学 | 构图、光线基本合理，达到 Gemini 2.5 Flash Image 基础水平 |

**可行性评级：⭐⭐⭐⭐⭐ 高**（已有成熟 API 可直接调用）

#### Skill IMG-D3：图生图（风格迁移/图片改写）

**一句话定义**
输入原始图片 + 改写指令，在保留主体结构的同时改变风格或内容细节。

**技术路线**
多模态模型 → GPT-4o Image（图生图） / Stable Diffusion img2img

**输入**
```
- 原始图片（上传）
- 改写指令（如：「改成扁平插画风格」、「换成深色背景」、「去掉背景只保留主体」）
- 改变强度（可选，0-100%，默认 50%）
```

**输出**
```
- 改写后的图片
- 自动展示前后对比
```

**质量标准**
| 维度 | 标准 |
|------|------|
| 结构保持度 | 原图主体布局保留，改变幅度符合强度设定 |
| 指令执行率 | 改写指令关键要素执行率 ≥ 85% |

**边界**
- ✗ 不做精确 Inpainting（局部修改 → 独立 Skill）
- ✗ 不保证高度相似的风格一致性（品牌风格锁定 → 另开 Skill）

**可行性评级：⭐⭐⭐⭐ 较高**

#### Skill IMG-E1：文案驱动配图生成

**一句话定义**
输入一段文案或标题，自动选择最优图片生成方式（SVG / 文生图 / 图库检索），产出匹配的配图。

**技术路线**
意图分析层 → 三路并行评估 → 选优 → 路由到子 Skill

**意图路由逻辑**
```
输入文案
    ↓
意图分析（LLM）
    ├── 需要图形化/图标型 → C1（SVG生成）
    ├── 需要写实/摄影风 → D1（文生图）
    └── 标准商业配图 → 图库检索（Unsplash/Pexels API）
    ↓
多方案展示 → 用户选择
```

**输入**
```
- 文案内容（1段，≤500字）
- 配图位置描述（可选，如：「文章头图」、「段落插图」）
- 风格偏好（可选）
```

**输出**
```
- 3个候选方案（不同技术路线各出1个）
- 每个方案：图片预览 + 生成方式说明 + 可编辑性说明
```

**质量标准**
| 维度 | 标准 |
|------|------|
| 语义相关性 | 配图与文案主题语义匹配，人工判断 ≥ 80% 合格 |
| 方案多样性 | 3个方案风格明显不同 |

**可行性评级：⭐⭐⭐⭐ 较高**（路由层需要一定工程化，但技术可行）

#### Skill VID-G1：Lottie JSON 动效生成

**输入**

```
- 色板（可选）

```

**输出**
```
- .json 文件（标准 Lottie 格式）
- 可直接在 LottieFiles Player 预览
- 可用 lottie-web / react-lottie 渲染到 Web
- 可用 lottie-android / lottie-ios 渲染到 App
```

**质量标准（竞品对标：LottieFiles AI + OmniLottie 论文基准）**
| 维度 | 标准 |
|------|------|
| 格式合法性 | Lottie JSON schema 验证通过，Player 可渲染 |

**边界**

**当前状态**

#### Skill EVAL-M1：UI 可用性评测

**一句话定义**
输入 UI 截图或代码，自动评估可用性问题（布局层级、交互合理性、信息密度），输出结构化评测报告。

**技术路线**
多模态视觉理解（GPT-4o Vision / Gemini Pro Vision）→ 规则引擎 + LLM 评判 → 报告生成

**输入**
```
- UI 截图（支持移动端 / PC / H5）
- 评测维度（可选，默认全维度）
- 对比版本（可选，用于前后对比）
```

**输出**
```
- 评测报告（Markdown / HTML）
- 问题列表（按严重级别分类：P0/P1/P2）
- 每个问题：截图定位标注 + 问题描述 + 改进建议
- 综合评分（0-100）
```

**评测维度（来自 TRAIL 模型 + 内部标准）**
| 维度 | 权重 | 说明 |
|------|------|------|
| 信息层级清晰度 | 25% | 主次信息是否分明 |
| 视觉引导性 | 20% | 用户视线流是否合理 |
| 交互一致性 | 20% | 同类操作反馈是否统一 |
| 内容可读性 | 20% | 字号/对比度/行距是否合规 |
| 响应式适配 | 15% | 不同屏幕尺寸下的表现 |

**质量标准（对标内部 TRAIL 模型 + Nielsen 十大可用性原则）**

**可行性评级：⭐⭐⭐⭐⭐ 高**（多模态 LLM 视觉理解已成熟）

#### Skill EVAL-N1：美学质量评测（AQE）

**一句话定义**
输入设计稿/图片，从视觉美学角度自动评分，输出 AQE（Aesthetic Quality Evaluation）报告，可用于筛选生成图片质量。

**技术路线**
多模态视觉模型 → AQE 多维度打分 → 报告生成

**评测维度（AQE 矩阵，来自内部框架）**
| 维度 | 权重 | 说明 |
|------|------|------|
| 视觉和谐度 | 30% | 色彩搭配、明暗对比是否协调 |
| 构图平衡性 | 25% | 视觉重心、留白是否均衡 |
| 风格一致性 | 20% | 全图元素风格是否统一 |
| 细节精致度 | 15% | 边缘处理、材质质感等微观质量 |
| 品牌契合度 | 10% | 与品牌调性的匹配程度（有品牌参考时生效） |

**输入**
```
- 图片文件（支持 PNG / JPG / SVG）
- 品牌风格参考（可选）
- 对比基准（可选，如竞品截图 / 历史版本）
```

**输出**
```
- AQE 综合评分（0-10，≥7 为合格）
- 各维度分项得分
- 优势描述 + 改进点
- 适用于：生成图片质量筛选、设计方案评审、A/B 测试支撑
```

**与生成 Skill 的联动**
```
IMG-C1 SVG生成 → EVAL-N1 美学评测 → 不达标 → 触发重新生成
```

**可行性评级：⭐⭐⭐⭐⭐ 高**（内部已有 AQE 框架基础，直接落地）

#### Skill EVAL-M2：视觉规范合规评测（Design Token）

**一句话定义**

**技术路线**
颜色/字体/间距提取 → 与 Design Token 规范库比对 → 差异报告

**输入**

**输出**

```
- 修复建议（附规范值）
- 合规率百分比
```

**质量标准**
| 维度 | 标准 |
|------|------|
| 检出率 | 主要规范违规检出率 ≥ 90% |
| 误报率 | 误报率 ≤ 10% |
| 处理速度 | 单页评测 ≤ 30 秒 |

**可行性评级：⭐⭐⭐⭐ 较高**（色彩/字体提取技术成熟，需规范库对接）

#### Skill EVAL-P1：自动化设计走查

**一句话定义**
输入设计稿 + 走查规则清单，自动执行设计走查并生成走查报告，替代人工逐项核查。

**技术路线**
规则引擎 + 多模态视觉理解 → 走查项目自动执行 → 报告生成

**走查规则体系**
```
走查规则类型：
├── 视觉规范类（颜色/字体/间距 → 对应 EVAL-M2）
├── 可用性类（交互流程/空状态/错误处理 → 对应 EVAL-M1）
├── 无障碍类（色彩对比度/字号/点击区域大小）
└── 业务逻辑类（流程完整性/边界状态覆盖）
```

**输入**
```
- 设计稿（Figma 链接 / 截图合集）
- 走查规则（选择预设规则集 / 自定义规则）
- 走查范围（指定页面 / 全量走查）
```

**输出**
```
- 走查报告（分类汇总 + 逐条详情）
- 通过/未通过统计
- 可导出为 Excel / PDF
- 预留 People-in-Loop 人工复核接口
```

**关键机制：People-in-Loop**
> 走查 Skill 不能完全替代人工判断。对于「模糊」的评测项（AI 置信度 < 70%），自动触发人工复核流程，由认证走查员（内部晋级标准达到某级别以上的设计师）进行确认。

**可行性评级：⭐⭐⭐⭐ 较高**

#### Skill EVAL-P2：人工辅助走查（People-in-Loop）

**一句话定义**
作为 EVAL-P1 自动化走查的补充，对 AI 置信度不足的模糊评测项，引入认证设计师进行人工判断和最终裁定，形成「AI走查 + 人工兜底」的闭环机制。

**设计理念**
> 走查 Skill 不能完全替代人工判断。P2 的核心价值在于：明确「人工需要介入的边界」，避免 AI 产生错误置信。

**触发条件**
```
EVAL-P1 走查结束后，满足以下任一条件触发 P2：
├── AI 置信度 < 70% 的评测项
├── 涉及「品牌感知/审美主观性」的评测项
└── P0 级别问题（必须人工二次确认）
```

**People-in-Loop 角色定义**
| 角色 | 资质要求 | 职责 |
|------|---------|------|
| **认证走查员** | 内部晋级标准达到 P5+ 的设计师 | 复核模糊项，给出最终判断 |
| **AI 协助员** | 无门槛 | 使用 EVAL-P1 跑初步走查，标记需人工复核的项 |

**工作流**
```
Step 1: AI 走查（EVAL-P1）→ 输出初步报告 + 待复核清单
Step 2: 认证走查员 复核待复核清单（预计 15-30 min）
Step 3: 认证走查员 更新最终结论
Step 4: 生成含人工确认印章的最终报告
```

**输入**
```
- EVAL-P1 走查结果（含「待复核清单」）
- 认证走查员账号（用于签署确认）
```

**输出**
```
- 附有认证签署的最终走查报告
- 区分「AI 判断」和「人工判断」两类结论
- 问题严重性最终裁定
```

**可行性评级：⭐⭐⭐⭐ 较高**（流程机制清晰，需结合内部工具实现分发和签署）

#### Skill EVAL-Q1：生成-评测自进化流水线

**一句话定义**
将「生成 Skill」和「评测 Skill」串联成自进化闭环：生成 → 评测打分 → 差距分析 → 变异重试 → 再评测，循环直至达到质量阈值，实现 Skill 自动优化。

**核心理念（来自 InfoVis-AutoEvo 框架）**
> 评测不只是终态质量检查，更是生成过程的反馈信号。当评分不达标时，自动分析原因并变异生成策略，驱动 Skill 持续进化。

**技术路线**
```
生成 Skill（任意方向）
    ↓ 输出结果
评测 Skill（对应方向）
    ↓ 评分 + 差距报告
差异计算层
    ├── 达标（≥ 阈值）→ 输出 ✅
    └── 未达标 → 变异策略生成（调整 Prompt / 参数 / 路由）
            ↓
        再次生成（最多 N 轮）
            ↓
        最终输出（最高分版本）
```

**联动 Skill 对**
| 生成 Skill | 评测 Skill | 阈值 |
|-----------|-----------|------|
| IMG-C1 SVG插画 | EVAL-N1 AQE | AQE ≥ 7.0 |
| IV-A5 信息图海报 | EVAL-N1 + EVAL-M1 | AQE ≥ 7.0 + 可用性 ≥ 80 |
| DOC-J1 PPT美化 | EVAL-O1 内容结构 | 结构评分 ≥ 75 |
| UI-B1 控制台 | EVAL-M1 可用性 | 可用性 ≥ 85 |

**输入**
```
- 生成任务描述（同对应生成 Skill）
- 质量阈值（可选，默认使用各 Skill 预设阈值）
- 最大迭代轮数（可选，默认 3 轮）
```

**输出**
```
- 最终达标版本（或 N 轮后的最高分版本）
- 迭代过程日志（每轮评分 + 变异策略说明）
- 质量进化曲线（可视化）
```

**边界**
- ✗ 不是无限循环（最多 N 轮，防止资源消耗失控）
- ✗ 评测 Skill 本身的误差会传导（Garbage In Garbage Out）
- ✗ 初期需要人工设定合理阈值

**可行性评级：⭐⭐⭐ 中等**（单步流水线可 2 天 PoC，全量覆盖需逐步扩展）
