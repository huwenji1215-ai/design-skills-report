🎨 方向二：图片生成 8 个 Skill · 4 技术路线
📐 SUB-CLASS ARCHITECTURE — 各子类的关系与层次
入口层 · 按用途触发
E 文案驱动型 → 上层封装，按意图路由
↓ 路由决策：意图分析选择最合适的生成方式
C 类 · 代码直出型
IMG-C1 SVG插画
LLM直接输出SVG代码
✓ 可编辑矢量
✓ 无需外部模型
适合：插画/图标/信息图
D 类 · 模型调用型
D1 文生图（通用）
D2 文生图（品牌锁定）
D3 图生图（风格迁移）
D4 局部重绘
调用外部图像模型
✓ 写实/高质量
✗ 不可直接编辑
适合：配图/营销/改图
F 类 · 加工裂变型
IMG-F1 多尺寸适配
基于已有图片加工
✓ 保持内容不变
✓ 批量多尺寸
适合：运营多平台发布
E类（文案驱动）不是独立生成器，而是调度层：分析文案意图 → 路由到 C/D/图库 → 返回候选方案。是C+D的上层封装。
C 类 vs D 类 — 同是「生成图片」，底层完全不同
C 类（代码直出）
输出物：SVG 代码文件
可编辑：✅ 直接改代码
依赖：无外部模型
适合：设计稿插图
D 类（模型调用）
输出物：PNG/JPG 图片
可编辑：❌ 像素不可改
依赖：外部图像模型
适合：写实配图
E 类 vs C/D 类 — 出发点完全不同
C/D 类：「我要生成一张图」→ 知道要用什么技术，直接调用对应 Skill
E 类：「我有一段文案，帮我配图」→ 不关心技术，让 Skill 自动决策路由
💡 类比：C/D = 点菜，E = 告诉服务员「我今天想吃点清淡的」，由服务员帮你选菜
F 类 — 独立于生成，属于加工处理
F 类的输入是已有图片（不是从零生成），核心是尺寸/裁切/扩展等加工操作。
与 C/D/E 不是同一个需求链，是独立的「图片后处理」路线。
