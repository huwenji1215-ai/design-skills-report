# 设计侧内容飞轮：构建 AI 驱动的生成-评审自进化引擎

## 前言

当设计团队开始把 AI 引入内容生产流程后，一个常见的困境是：**AI 生成的内容质量参差不齐，而评审标准又高度依赖人的经验判断**。

每次生成一张可视化图表、一份 PPT、一页信息图，设计师要重新：

- 盯着屏幕判断图表类型对不对
- 凭感觉评估信息是否在 5 秒内可读
- 凭经验发现视觉噪声在哪里
- 给出"重做"或"还行"的结论，但说不清为什么

这意味着，设计评审的标准存在于人的脑子里，而不是系统里。**它无法被复用、无法被量化、无法被传递给 AI 生成器**。

这篇文章记录了我们对这个问题的思考，以及一套基于 Skill 机制的解决方案——我们称之为**「设计侧内容飞轮」**。

---

## 一、问题的提出：生成和评审为什么是断裂的？

### 1.1 生成能力的本质缺陷

AI 生成器（无论是 visual-creator 还是任何图表生成工具）本质上是一个**单向执行器**：

```
[用户输入需求] → [AI 生成内容] → [输出产物]
```

它不知道自己生成的东西好不好。它没有"质量感知"能力——只会生成，不会判断。

更关键的是：**每次生成都是独立的**。上一次生成发现的问题（比如总是遗漏阈值线、图表类型选错），对下一次生成毫无约束力。失败经验不会被沉淀，错误会一直重复。

### 1.2 评审能力的本质缺陷

AI 评审器（或人工评审）存在对称的问题：**它能发现问题，但结论无法自动反哺给生成器**。

传统评审流程：

```
[生成产物] → [评审发现问题] → [人工修改] → [再次评审]
```

评审结论停在评审报告里。设计师看到"核心洞察不可读"的反馈，需要自己理解、自己修改——这个"翻译"过程依赖个人能力，且无法规模化。

### 1.3 核心矛盾：没有闭环，就没有进化

生成器不懂评审标准，评审器不驱动生成器改进。**两个能力之间缺少一条自动化的信息回路。**

结果就是：AI 辅助生产了大量内容，但内容质量的提升速度极慢——每次都在重复类似的错误，团队的设计判断力没有被系统化积累。

## 二、问题的分析：什么是「内容飞轮」

### 2.1 飞轮的定义

「飞轮」这个词来自商业策略——**每一次转动都为下一次转动积蓄势能，随着旋转次数增加，驱动力越来越强**。

在 AI 内容生产的语境里，「内容飞轮」的含义是：

> **AI 生成内容 → 评测引擎量化质量 → 差异结论反哺生成器 → 带着约束重新生成 → 再次评测……循环迭代，直到达到质量标准。**

```
[输入: 数据 + 意图]
       ↓
  [生成阶段]  ←─────────────────────────────┐
       ↓                                    │
  [评测阶段]  → 分数 ≥ 通过阈值 → ✅ 输出    │
       ↓                                    │
  [差异计算]                                │
       ↓                                    │
  [变异反哺]  → 将失分原因注入生成器指令 ───┘
```

这是一个**真正的闭环**：每一轮迭代都让生成器更了解"什么是好的内容"，直到产物达标为止。

### 2.2 飞轮需要回答的三个核心问题

**问题一：用什么标准评测？**

评测标准必须是结构化的、可量化的，而不是"感觉不错"或"还差点意思"。

我们设计了四个通用维度，适用于所有设计内容类型：

| 维度 | 核心问题 | 权重 |
|------|---------|------|
| 可读性 & 信息层级 | 核心信息能否在 3 秒内被识别？ | 高 |
| 受众对齐 | 设计语言是否匹配目标受众？ | 中 |
| 认知负担 | 提取信息需要多少认知资源？ | 高 |
| 视觉一致性 | 整体是否有统一的设计语言？ | 中 |

针对图表类型，还额外叠加三维专项评测：信息路由（图表类型是否匹配意图）、盲测解码（不看上下文能否提取洞察）、认知摩擦力（噪声源量化）。

**问题二：如何把评测结论「翻译」给生成器？**

评测报告不能只是一份人读的文档。它必须被结构化处理，转化成生成器能直接执行的「变异补丁」：

```
❌ 不可用的评测结论: "图表不够清晰"

✅ 可执行的变异补丁:
   [P0] 缺少阈值线 → 指令: 在 y=3.5 处添加橙色虚线，标注「预警线 3.5」
   [P0] 同色无预警区分 → 指令: 低于 3.5 的柱子改为红色 #ef4444，高于 3.5 改为绿色 #22c55e
   [P1] 缺少差距量化 → 指令: 预警柱子上标注「差 -0.7」等具体差值
```

每条补丁都对应一个具体失分维度、一个可操作的修复指令、以及修复后预期的分数提升。

**问题三：如何防止反复犯同样的错误？**

仅靠单轮反馈不够。系统需要**跨轮次积累进化知识**：

- `known_failures`：本 case 已确认的设计禁忌，禁止重复
- `design_constraints`：有效修复沉淀成的硬性约束，强制保持
- `type_traps`：特定意图下容易误选的图表类型，写入警告

这些知识在每次迭代结束后自动更新，注入下一轮生成的 Prompt 约束层。

## 三、解决方案：设计「InfoVis 元技能自进化引擎」

### 3.1 整体架构

我们把这套飞轮机制封装为 `infovis-autoevo` Skill——一个将 `visual-creator`（生成）和 `infovis-evaluator`（评测）串联起来的**元技能**。

```
系统流水线: 生成阶段 → 多维交叉评测 → 差异计算 (Diff) → 技能变异 (Mutation)

[输入层_INPUT]
  原始数据负载 (Payload) + 意图配置 (Config)
         ↓
[生成层_PIPELINE]
  Meta_Router → Generator_Skill.v1.0 → 视觉产物 (Visual_Output)
         ↓
[评测层_MATRIX]
  技能A: 信息路由 (LLM)  ─→ Expected Topology
  技能B: 盲测解码 (VLM)  ─→ Perceived Topology
  技能C: 认知摩擦力      ─→ Friction Score
                          ↓
                   拓扑差异计算引擎
                   → PASS / FAIL
         ↓ FAIL
[进化层_MUTATION]
  Mutator_Skill → 差异报告 → 自动重写生成器指令
         ↓
  [返回生成层，开始下一轮迭代]
         ↓ PASS (分数 ≥ 9.6)
  输出最终可视化产物 + 完整迭代报告
```

这个架构对应了文章开头提到的断裂：**生成和评审之间，有了 Diff Engine 和 Mutator 这两个「翻译层」**。

### 3.2 评测引擎设计（三维交叉矩阵）

评测不是单一打分，而是三个独立视角的交叉验证：

**技能 A — 信息路由（语义判断层）**

从数据意图出发，推导「期望拓扑」（Expected Topology）——这个数据和意图的组合，理论上应该长什么样？应该包含哪些必要元素？

```json
{
  "chart_type": "BarChart",
  "core_insight": "评分 < 3.5 的部门触发预警",
  "required_elements": ["threshold_line", "alert_color_encoding", "value_labels"]
}
```

**技能 B — 盲测解码（视觉感知层）**

把图表代码/截图单独给一个「不知道数据背景」的观察者，让它描述：这张图里，你能在 5 秒内读出什么？核心洞察在哪里？

这模拟了真实读者的第一反应，测量图表的「自解释能力」。

**技能 C — 认知摩擦力（噪声量化层）**

扫描视觉元素，量化认知干扰：背景网格线密度、颜色对比度、信息密度、装饰性元素占比……

用数字说清楚「为什么感觉乱」。

**拓扑差异计算（Diff Engine）**

三个技能的输出汇聚到 Diff Engine，对比期望 vs 感知，计算综合分：

| 维度 | 权重 |
|------|------|
| 图表类型匹配 | 20% |
| 核心洞察可读性 | 35% |
| 必要元素覆盖率 | 25% |
| 认知摩擦力惩罚 | 20% |

**通过阈值：9.6 分**（有意设置得很高，倒逼多轮迭代直到真正高质量）

### 3.3 变异反哺协议（Mutation Protocol）

当分数低于 9.6，进入变异阶段。变异补丁按优先级分为三级：

| 级别 | 触发条件 | 生成约束强度 |
|------|---------|------------|
| P0（必须修复） | 单维度失分 > 2.0 | 用【必须】强调，不修复无法收敛 |
| P1（强烈建议） | 单维度失分 1.0~2.0 | 用【强烈建议】强调 |
| P2（锦上添花） | 单维度失分 < 1.0 | 用【建议】标注 |

补丁内容不只是问题描述，而是**可直接注入 Prompt 的指令**：

```
=== 自进化约束包 (v2) ===

【P0 - 必须修复】
1. [TYPE] 将图表类型从 RadarChart 改为 BarChart，
   原因：「高管汇报-预警聚焦」意图下 BarChart 更直观

2. [INSIGHT] 添加 threshold_line at y=3.5（橙色虚线），
   添加 alert_color_encoding：低于阈值用红色 #ef4444 标注

【已知禁止重复的错误】
- 禁止：所有柱子同色（掩盖预警状态）
- 禁止：缺少阈值线（读者无法判断哪些部门有问题）

【保持的成功约束】
- 保持：底部统计摘要「N/5部门预警」
- 保持：预警柱子顶部的「⚠ 预警」标签
```

### 3.4 激进变异策略（防止局部最优）

当连续 2 轮分数提升 < 0.1，说明当前设计方向已经接近局部最优，需要突破性改变：

```
[激进变异触发]
→ 丢弃所有历史补丁
→ 从 Expected Topology 重新推导
→ 完全重新选择图表类型
→ Prompt 注入："请完全重新设计，忽略之前的所有版本"
```

## 四、实战验证：一张电商部门评分预警图的自进化过程

理论设计完成后，用一个真实 case 来验证整套机制。

### 4.1 Case 输入

```json
{
  "data_payload": {
    "departments": ["电商", "物流", "客服", "运营", "技术"],
    "scores": [2.8, 4.1, 3.2, 4.7, 3.9],
    "threshold": 3.5,
    "period": "2024-Q1"
  },
  "viewer_intent": "高管汇报 - 预警聚焦",
  "audience": "非技术决策者",
  "pass_threshold": 9.6
}
```

### 4.2 第 1 轮：初始生成 → 5.35 分

**生成器输出**：一张普通 BarChart，所有柱子统一蓝色，无阈值线，无预警标识。

**评测结果**：

- 技能 B 盲测：「所有柱子同色，无法在 5 秒内识别哪些部门有问题」
- 技能 A 路由：缺少 threshold_line、alert_color_encoding
- **综合分：5.35 / 10.0 → ❌ FAIL**

**变异补丁生成**：

```
🔧 [P0-INSIGHT] 缺少预警阈值线（最大失分维度）
   指令: 添加 y=3.5 橙色虚线，标注「预警线 3.5」

🔧 [P0-COLOR] 同色无预警区分
   指令: 低于3.5 → 红色 #ef4444，高于3.5 → 绿色 #22c55e

🔧 [P1-ELEMENT] 缺少预警标识
   指令: 预警柱子顶部添加「⚠ 预警」标签
```

### 4.3 第 2 轮：注入补丁 → 8.30 分

**生成器输出**（携带第 1 轮补丁）：加入了阈值线、双色编码、预警标签。

**评测结果**：

- 技能 B 盲测：「可以识别预警状态，但不知道具体差了多少」
- 技能 A 路由：缺少差距量化标注、底部统计摘要
- **综合分：8.30 / 10.0 → ⚠️ WARN，仍未达标**

**变异补丁生成**：

```
🔧 [P1-INSIGHT] 缺少与阈值的差距量化
   指令: 预警柱标注「差 -0.7」/「差 -0.3」

🔧 [P1-ELEMENT] 缺少底部统计摘要
   指令: 添加「2/5部门预警」汇总卡片

🔧 [P2-FRICTION] 柱子渐变色引入轻微噪声
   指令: 改为纯色填充
```

### 4.4 第 3 轮：注入全部补丁 → 9.60 分 ✅

**生成器输出**（携带前两轮所有补丁+已知禁止项）：

- ✅ BarChart 类型正确
- ✅ 3.5 预警线（橙色虚线）
- ✅ 红/绿双色编码（预警/正常）
- ✅ 预警柱顶部「⚠ 预警」标签
- ✅ 差距量化标注（差 -0.7 / 差 -0.3）
- ✅ 底部统计摘要（2/5部门预警 · 3/5部门达标 · 均分3.74）

**评测结果**：

- 技能 B 盲测：「5秒内清晰看出电商/客服预警，整体状态一目了然」
- readability_score: 0.97
- **综合分：9.60 / 10.0 → ✅ PASS**

### 4.5 自进化轨迹

| 轮次 | 得分 | 状态 | 核心改进 |
|------|------|------|---------|
| 第 1 轮 | 5.35 | ❌ FAIL | 初始版本，无预警标识 |
| 第 2 轮 | 8.30 | ⚠️ WARN | 加入阈值线+双色编码 |
| 第 3 轮 | 9.60 | ✅ PASS | 差距量化+统计摘要 |

**3 轮迭代，+4.25 分，从「看不出问题在哪」到「5秒决策级」。**

## 五、架构扩展：设计评测的通用化

### 5.1 从「图表专项」到「设计通评」

InfoVis 飞轮验证了这套机制的可行性后，我们把评测能力向上抽象，扩展为 `design-evaluator`——**适用于任意视觉传达产物的通用评测 Skill**。

```
[输入] 任意设计产物
   ↓
[Layer 0] 内容类型识别（图表/PPT/报告/海报/信息图）
   ↓
[Layer 1] 共同维度评测（可读性·受众对齐·认知负担·视觉一致性）
   ↓
[Layer 2] 类型特化子评测器
   ├── InfoVis → infovis-evaluator（三维矩阵）
   ├── PPT    → 叙事结构 + 单页原则 + 演讲适配
   ├── 报告   → 摘要质量 + 结构导航 + 数据呈现
   └── 海报   → 瞬时吸引力 + 信息路径 + 品牌一致性
   ↓
[Layer 3] 综合评分 + 变异建议
   ↓
[Layer 4] 自进化引擎（可选）
```

不同内容类型有不同的通过阈值：图表类要求 9.6 分（严苛），PPT/报告要求 8.5 分（标准）。

### 5.2 使用场景矩阵

| 场景 | 触发方式 | 适用内容类型 | 典型用途 |
|------|---------|------------|---------|
| **单次评测** | 「评测这个图表」 | 所有类型 | 快速诊断设计问题 |
| **自进化优化** | 「帮我迭代到合格」 | 所有类型 | 批量提升内容质量 |
| **批量 Case 验证** | 提供多个 Case | 主要图表类 | 新模板上线前质检 |
| **快速反馈** | 「快速看一下」 | 所有类型 | Top 3 问题快速扫描 |

### 5.3 关键设计原则

**原则一：评测标准要「可执行化」**

评测结论不能停在"不够清晰"这种层面。每条结论必须对应一个具体的修复指令，能直接被 AI 生成器或设计师执行。这是飞轮能转起来的前提。

**原则二：通过阈值要「倒逼」**

9.6 分的阈值不是随意设置的。它足够高，确保系统不会在「勉强合格」时停下来，而是持续迭代到真正高质量。低阈值会让飞轮失去驱动力。

**原则三：失败知识要「永久积累」**

`known_failures` 列表是整套机制的记忆核心。每一次失分都是一次学习，这个学习要被固化成约束，写入下一轮生成的前置条件里。失败不可重复，是飞轮加速的关键。

**原则四：激进变异防止「局部陷阱」**

当连续迭代收益趋近于零，说明当前方向已经撞到天花板。这时候需要的不是微调，而是重新出发。激进变异机制是防止飞轮在局部最优里空转的安全阀。

## 六、当前局限性

**生成器集成深度有限**：目前自进化主要通过 Prompt 注入约束来驱动生成器改进，而不是真正修改生成器的权重。这意味着复杂的结构性问题（比如数据选型根本错误）可能需要超过 5 轮才能收敛。

**评测依赖 LLM 主观判断**：技能 B（盲测解码）和维度 D2（受众对齐）部分依赖大模型的主观判断，评分在不同运行间存在一定波动。对严苛场景，建议运行 2-3 次取均值。

**批量场景尚未打通**：目前自进化是单 Case 串行的，批量 Case 回归测试的并行化还在开发中。

**人工确认节点**：当前内容修改仍需用户确认（非 yolo 模式）。完全自动化流水线适合 CLI 批量执行场景。

---

## 七、结语

内容飞轮的本质，是让 AI 内容生产从「用完即忘」变成「越用越准」。

这不是一个模型能力的提升，而是一个系统工程实践：通过合理设计生成-评测-反哺的闭环，让每一次内容迭代都为下一次积累势能。

在实际应用中，随着 `known_failures` 和 `design_constraints` 的积累，飞轮转到后期的提速是显著的——第 1 轮需要修复 3 个 P0 级问题，第 2 轮只剩 2 个 P1，第 3 轮往往只需微调就能过关。

**这不是因为 AI 变聪明了，而是因为失败被系统性地记住了。**

失败会终止，洞察会留下，飞轮会加速。

       ↓
