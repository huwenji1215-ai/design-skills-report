KF-001 · AQE-E · DIR-01 · UI生成
代码过度压缩时 WCAG 对比度验证标注被省略
失败类型
AQE-E（可访问性）× DIR-01 UI 生成 × Kimi K2.5
触发条件
模型追求代码简洁（<100行），以「轻量化」为由省略 CSS 变量中的 WCAG 对比度注释
症状
Model B（Kimi）：无对比度注释，AQE-E=6.0，选中状态仅靠颜色区分，无非颜色辅助 → FAIL 主因
根因
Skill description 虽有 WCAG AA 要求，但未强制约束「无论代码长短必须在 CSS 变量中注释对比度值」
修复指令
description 增加：「无论代码长短，主要文字色 CSS 变量必须注释对比度数值（如：/* 对比度 7.8:1 */）」
防重现规则
准入 Checklist 新增 P0：「主要文字色变量必须有对比度注释，缺失直接判定 AQE-E BLOCK」
KF-002 · AQE-B · DIR-01 · UI生成
consumer_app 情感类产品，「暖色调」描述过宽导致主色色温失准
失败类型
AQE-B（色彩美学）× DIR-01 × Gemini 3.1 Pro Preview
触发条件
用户指定「暖色调」但未精确约束色温范围；模型选用深砖红（#D8432A），符合「暖色」但偏警示/危险语义
症状
Model C 主色 #D8432A，与「情绪记录/疗愈」调性有张力，AQE-B 降为 8.3（对比 Model A 的 8.8）
根因
「暖色调」覆盖 Hue 0°-60°（红→橙→黄），范围过宽；情感类产品需精确到橙黄系（Hue 10°-50°）
修复指令
description 增加：「当产品类型为 consumer_app（情感/疗愈类），暖色调默认推荐橙系（#FF8A65~#FFB74D），避免红系（≥#E53935）」
防重现规则
准入 Checklist 新增 P1：「consumer_app 情感类产品的主色 Hue 范围应约束在 10°-50°（橙黄系），需在 description 中显式声明」
