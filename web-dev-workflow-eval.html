§2 · Mock 测试案例

评测方向

UI生成 · DIR-01

单页 HTML 网页开发场景

竞品基准

Claude Artifacts / v0.dev

HTML原型质量锚点

用户输入 Prompt

帮我做一个AI工具导航页（单页HTML） 要求： 1. 展示常见AI工具分类：文生图、代码助手、写作助手、视频生成等 2. 现代简洁风格，深色/浅色都可以 3. 响应式设计，在手机和电脑上都能看 4. 每个工具卡片有工具名、简介、进入按钮

System Prompt（Skill 规范注入）

输出物类型：单文件HTML（包含内联CSS和JS） 视觉层级：标题>分类>卡片>描述，字号体系清晰（≥3级字号） 色彩：主色系≤5个，使用协调配色，文字对比度≥4.5:1 布局：8px间距体系，留白充足，内容不超过页面70% 响应式：支持375px~1920px 交互：按钮有hover状态，可交互元素有focus样式

注：System Prompt 模拟了 web-dev-workflow 技能的规范约束，反映 Skill 实际向模型注入的质量要求。

§5 · 综合结论与改进建议
7.54

整体评级：WATCH

三个模型综合 AQE 均分 7.54，介于 PASS（7.5）和 WATCH（7.0-7.4）边界。2/3 模型单次达到 PASS 标准，但 AQE-B 色彩美学维度（均值 6.7）是最大短板，且 Claude-Opus-4.7 在视觉层级维度明显失分（5.5），表明 Skill 的色彩规范约束力不足，未能有效引导所有模型遵循色彩体系要求。

✅ 版式规范优秀 8.5 ✅ 风格一致性强 8.0 ⚠️ 色彩约束偏弱 6.7 ⚠️ 竞品对标缺失 P0 ❌ Claude层级不足 5.5

🏆 优势亮点

  • ✅ 踩坑文档体系完善：7个专项 references 文档，远超标准要求的3条
  • ✅ 工作流闭环设计：本地验证→部署→确认→发链接，流程规范严谨
  • ✅ 文件拆分机制：明确的行数决策表防止上下文超限
  • ✅ 元能力绑定：强制调用 meta-execution，质量保障有闭环
  • ✅ 版式约束有效：所有模型都使用8px间距体系和flex/grid布局

⚠️ Known Failures（已知失败）

AQE-B · UI生成 · 色彩约束力弱
根因：SKILL.md 未明确指定色彩上限（≤5色相）和情感调性要求
修复：在 SKILL.md 生成规范中添加「色彩约束」条款
AQE-A · 标题层级不稳定
根因：Skill 未明确要求≥3级字号体系，模型自由裁量
修复：在 System Prompt 规范中强制要求3级字号
P0 · 竞品对标锚点缺失
根因：SKILL.md 无竞品对标表
修复：补充「对标 v0.dev / Claude Artifacts」的竞品基准

🔧 优化建议

  1. 补充竞品对标表（P0优先）
    在 SKILL.md 中明确「对标 v0.dev（B端）/ Claude Artifacts（原型）」
  2. 加强色彩约束
    在生成规范中增加：主色系≤5个色相，强调色≤2个
  3. 明确字号体系要求
    要求生成 UI 必须包含 ≥3 级字号（建议28/18/14px）
  4. WCAG 对比度升级为强制
    将 checklist「可选」的对比度检查升级为 P0 强制项

结论：web-dev-workflow 在工作流设计、踩坑文档、部署规范方面表现优异，是一个工程质量扎实的 Skill。主要短板在于设计侧美学约束力度不足——色彩规范、竞品对标、字号体系要求未在 SKILL.md 中明确写入，导致模型输出质量参差不齐（Claude 失分 vs. Kimi/DeepSeek 通过）。建议按 Known Failures 中的修复建议迭代后，达到稳定 PASS 水准。
